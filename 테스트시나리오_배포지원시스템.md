## 시나리오 1. 아웃룩 동기화 (failed to fetch 해결 확인)
1. 대시보드 접속 → `☁️ 아웃룩 동기화` 클릭
2. **기대:** "☁️ 아웃룩 동기화 완료 — N개 일정" 토스트, 그리드에 일정 표시
3. **합격:** failed to fetch 없이 일정이 채워짐
   - 실패 시 → 프록시 끄고 재시도 (시나리오상 네트워크 이슈 분리)

## 시나리오 2. 태깅 분류 정확성 (핵심)
아래 일정을 아웃룩에 등록 후 동기화하거나, 대시보드에서 직접 추가해 확인.

| 등록 일정 | 기대 판정 |
|-----------|-----------|
| 하루종일 `[티켓] 처리` | ✅ 가능 (하루종일이어도 가능) |
| 하루종일 `[내부] 문서작업` | ✅ 가능 |
| 하루종일 `[내부업무] …` (기존 표기) | ✅ 가능 (동의어 인식) |
| 하루종일 `[기아] 현장` (고객사명) | ⛔ 불가 |
| 시간단위 `[티켓] 원격지원` | ⛔ 불가 (원격) |
| 태그 없는 `주간회의` | ✅ 가능 (회의·무태그 = 가능) |
| `연차` | ⛔ 불가 |

1. 돌림판 열기 → 해당 날짜/시간 선택
2. **합격:** 가용자 목록에 ✅인 사람만 "가능", ⛔는 "불가"로 표시

## 시나리오 3. 수동 분류 오버라이드 → Notion 공유 (★ 가장 중요)
> ✅ 오버라이드는 **Notion(SCHEDULES_DB `배포분류`)에 저장**되어 다른 운영자에게도 공유됩니다.

1. **브라우저 A(매니저):** 무태그 일정 클릭 → "배포지원 분류"를 `✅ 가능`으로 변경
2. **기대 A:** 해당 일정이 가용으로 바뀜 + Notion에 분류 저장
3. **브라우저 B(타인):** 같은 날짜 새로고침
4. **합격:** **B에서도** 그 일정이 `가능`으로 보임 (= 다른 사람에게도 반영)
   - 반대로 `⛔ 불가`/`⚠️ 긴급` 변경도 B에 반영되는지 확인

## 시나리오 4. 라운드로빈 공정성 + 긴급(⚠️)
1. 돌림판에서 스핀 → 당첨자 확인 → **재스핀 3회**
2. **합격:** 재스핀해도 **같은 사람 유지** (공정성 잠금)
3. 가능 태그 + `[긴급]` 일정 있는 사람 → 가용자 목록에 **⚠️ 배지** 표시
4. **합격:** ⚠️ 보고 매니저가 제외/투입 판단 가능
5. 배정 확정 → 같은 인원이 **연속 당첨되지 않고** rrOrder 후순위로 밀리는지 확인

## 시나리오 5. 배정 end-to-end
1. 돌림판 당첨 → `🚀 이 멤버로 배포지원 일정 투입` → 배포 모달 저장
2. **합격:** 그리드에 빨간 배포 블록 + 담당자 메일창(mailto) 열림 + Notion 일정/스핀이력 저장
3. (선택) 배포 일정 삭제 → 해당 인원 우선순위 복구되는지 확인
