# 🚀 배포지원 투입관리 대시보드 사용가이드

**접속 주소:** https://emitlight.github.io/deploy-schedule/

---

## 처음 접속했을 때

페이지에 처음 들어오면 **아무 일정도 없는 빈 달력**이 보입니다. 정상입니다.
아래 순서대로 한 번만 설정하면 이후에는 자동으로 데이터가 채워집니다.

---

## 1단계 — Worker URL 연결 (최초 1회)

> 이 설정은 처음 한 번만 하면 됩니다. 브라우저에 저장되어 이후엔 자동 적용됩니다.

1. 헤더의 **☁️ 아웃룩 동기화** 버튼 클릭
2. 팝업이 뜨면 아래 URL을 그대로 복사해서 붙여넣기 후 확인

```
https://deploycalendar.hayoung-lee.workers.dev
```

3. **아웃룩 동기화** 버튼 <u>한 번 더</u> 클릭
4. **동기화가 "failed to fetch"로 실패하면** = 사내망(SHIELD Gate) 프록시가 막은 것입니다.
   → 윈도우 설정 > 프록시 > 설정 스크립트 사용 **'끔'** → 동기화 완료 후 다시 **'켬'**
   → (근본 해결은 IT에 `deploycalendar.hayoung-lee.workers.dev` 화이트리스트 요청)

---

## 2단계 — 아웃룩 일정 불러오기

> 아웃룩 일정은 매일 **오전 7시 · 낮 12시 30분 · 오후 3시**에 자동 동기화됩니다.
> 처음 접속했을 때는 수동으로 한 번 불러와야 합니다.

헤더의 **☁️ 아웃룩 동기화** 클릭 → 잠시 기다리면 8명의 일정이 달력에 표시됩니다.

---

## ⭐ 가장 중요 — 아웃룩 일정 태깅 규칙

이 시스템은 **아웃룩 일정 제목의 `[태그]`로 "배포지원에 빠져도 되는 사람"을 판별**합니다.
**제목 맨 앞에 아래 태그를 붙여주세요.** (헤더 **🏷️ 태그 안내** 버튼에서도 확인 가능)

### 🟢 배포지원 "가능" (진행 중이지만 빼도 되는 일)
| 태그 | 의미 | 예시 |
|------|------|------|
| `[티켓]` | CRM 티켓 처리 | `[티켓] TT119590 처리` |
| `[파트너]` | 파트너사 지원 | `[파트너] A사 문의 대응` |
| `[내부]` | 내부 업무 | `[내부] 문서 작성` |
| `[교육]` | 교육 수강·준비 | `[교육] 보안 교육` |
| `[대기]` | 여유·대기 | `[대기]` |

### 🔴 배포지원 "불가" (빼면 안 되는 일)
| 태그 | 의미 |
|------|------|
| `[고객사]` | 고객사 현장·전담 (예: `[고객사] 기아 현장지원`) |
| `[외근]` · `[출장]` | 외근·출장 |
| 연차·반차·휴가 등 | 휴가류 (태그 없어도 자동 인식) |

### ⚠️ 긴급 (가능 업무지만 빼지 말아야 할 때)
가능 태그 + **`[긴급]`** → 풀에는 포함되지만 가용자 목록에 **⚠️** 로 표시되어 운영자가 신중히 판단합니다. (예: `[티켓][긴급] 장애 대응 중`)

### 꼭 기억할 5가지
1. **회의·미팅·태그 없는 일정은 기본 "가능"** 입니다. **빼면 안 되는 일**(고객사 현장·외근·출장·휴가)만 🔴태그나 휴가로 표시하세요.
2. **하루종일 일정이어도** `[티켓]`·`[내부]` 등 가능 태그면 **배포지원 가능**으로 잡힙니다.
3. **가능 키워드 없는 `[태그]`는 모두 "불가"** — 고객사명(`[기아]`·`[국민카드]`)·프로젝트명 등 `[티켓]`·`[내부]` 같은 가능 키워드가 없는 대괄호 태그는 전담 업무로 보고 제외됩니다.
4. **태그 형식은 유연합니다** — `[티켓]` · `[티켓] 한국산업진흥원` · `[티켓_한국산업진흥원]` · `[완료/티켓]` · `[내부/LMS]` 모두 인식됩니다 (대괄호 안에 `티켓·파트너·내부·교육·대기`가 `/`·`_`·공백으로 구분돼 들어가면 OK).
5. `[내부업무]` 는 `[내부]` 와 같게 인식됩니다.

## 3단계 — 화면 보는 법

| 색상 | 의미 |
|------|------|
| 🔵 파랑 | 회의·아웃룩 일정 |

| 🟤 갈색 | 기타 |

- **◀ ▶** 버튼으로 주 이동, **요일 탭** 클릭으로 해당 날짜만 보기
- **🟢 빈 시간 보기** 로 가능한 시간대 한눈에 확인
- 일정(아웃룩 포함)을 **클릭하면 상세 + 배포지원 분류 조정** 가능 (아래 "수동 분류 조정" 참고)

## 🎲 배포 담당자 뽑기 — 핵심 기능

### Step 1. 돌림판 열기
헤더의 **🎲 라운드로빈 돌림판** 클릭
> 모달이 화면보다 길면 모달 안에서 스크롤됩니다(작은 모니터 대응).

### Step 2. 배포 날짜·시간 설정
날짜·시간을 설정하면 **그 시간에 "배포지원 가능"한 사람만** 돌림판에 올라옵니다.
- 가능 판단 기준 = 위 **태깅 규칙** (회의·무태그·🟢태그는 가능, 🔴태그[고객사/외근/출장]·휴가만 제외)
- `[긴급]` 인 사람은 가용자 목록에 ⚠️ 로 표시됩니다.

### Step 3. 돌리기 — 공정성 보장
**🎲 돌리기!** 클릭 → 당첨자 발표 🎉
- 🔄 **재스핀해도 같은 사람으로 고정**됩니다 (한 번 뽑힌 결과를 임의로 바꾸지 않음 = 형평성).
- 모달을 닫았다 다시 열어도 **당첨자가 유지**됩니다.
- 배정이 확정되면 그 사람은 **자동으로 후순위로** 밀려, 다음엔 가장 오래 안 뽑힌 사람이 우선됩니다.

### Step 3-1. 당첨자 초기화 (다시 추첨)
당첨 후 나타나는 **✕ 당첨자 초기화 (다시 추첨)** 버튼을 누르면 초기화되고 다시 돌릴 수 있습니다.
> 누가 언제 초기화했는지 **Notion(SPINLOG_DB)에 `[초기화]` 로그로 기록**됩니다(공정성 감사용). 이 기록은 라운드로빈 순서에는 영향을 주지 않습니다.

### Step 4. 배포 일정 확정
당첨 후 **🚀 이 멤버로 배포지원 일정 투입하기** 클릭 → 폼 자동 채움 → 내용 입력 후 **투입 확정**
→ 달력에 🔴 표시 + Notion 저장 + **담당자에게 메일 작성창(mailto) 자동 열림** (발송은 직접)

## 🛠 수동 분류 조정 (운영자용)

구성원이 태그를 누락했거나 예외 상황일 때, 운영자가 직접 보정할 수 있습니다.

1. 달력에서 **해당 일정 클릭**
2. **"배포지원 분류"** 에서 선택: `🏷️ 자동(태그 기반)` / `✅ 가능` / `⛔ 불가` / `⚠️ 긴급`
3. 선택 즉시 저장됩니다.

> 이 분류는 **Notion(SCHEDULES_DB의 `배포분류` 속성)에 저장되어 다른 운영자에게도 동일하게 반영**됩니다. (아웃룩 원본 일정은 바뀌지 않습니다.)

## 일정 직접 추가
아웃룩에 없는 일정을 수동 등록할 수 있습니다.
1. 달력의 **빈 칸 클릭** → 일정 추가 팝업
2. 제목·시간·종류 입력 후 저장
3. Notion에 자동 저장 → 팀 전원이 바로 확인

## 자주 묻는 질문

**Q. 아웃룩 일정이 안 보여요.** → **☁️ 아웃룩 동기화** 클릭. "failed to fetch"면 1단계 4번(프록시) 참고.

**Q. 돌림판에 내가 안 나와요.** → 🔴태그(`[고객사]`·`[외근]`·`[출장]`)가 붙었거나 휴가/외근/출장일 때만 제외됩니다. (회의·무태그는 '가능'이라 제외 안 됨)

**Q. 일정이 있는데도 돌림판에 떠요.** → 정상입니다. 회의·미팅·무태그·🟢태그(`[티켓]`·`[내부]` 등)는 "빼도 되는 일"로 보기 때문입니다. 정말 빠지면 안 되면 🔴태그/휴가로 표시하세요.

**Q. 다른 사람 PC에서 다르게 보여요.** → 새로고침하면 Notion 최신 데이터를 불러옵니다.

## 배포지원 투입관리 DB (Notion)

링크: https://app.notion.com/p/DB-37596f61215f81579cc9fdcd23b98bf2?source=copy_link

- **SCHEDULES_DB** — 모든 일정 (아웃룩 동기화 + 직접 추가 + 배포투입)
  - `배포분류` — 운영자가 수동 지정한 분류(가능/불가/긴급). 비어 있으면 태그로 자동 판단.
- **SPINLOG_DB** — 라운드로빈 스핀 이력
  - 배정 확정 시 기록되며 "가장 오래 안 뽑힌 순" 계산에 사용됩니다.
  - `[초기화]` 항목 — 당첨자 초기화 이력(감사용, 순서 계산에는 미반영).
